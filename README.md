## About Carokann

Carokann은 탭 단위로 루틴과 반복 작업을 정리하고, 주기와 시간 기준에 따라 작업을 자동으로 다시 시작할 수 있게 만든 개인용 작업 리셋 트래커입니다.

이 프로젝트는 UI를 먼저 붙이기보다 탭/작업 상태 모델, 반복 작업 계산, 저장 전략을 먼저 굳힌 뒤 화면을 얹어가는 방식으로 발전했습니다. 이 README는 현재 코드, Git 커밋 이력, 그리고 2026년 3월 15일부터 2026년 3월 25일까지의 로컬 Codex 세션 로그를 함께 참고해 정리했습니다.

## 현재 구현 상태

- 완료된 범위: 탭/작업 CRUD, 반복 작업 주기 계산, 다중 체크, 드래그 앤 드롭 정렬, 삭제 후 복구, 로컬 저장, Google OAuth 로그인, 프로필 조회, `TrackerSnapshot` 기반 DB 저장, 로그인 UI, 기본 skeleton 로딩
- 현재 서비스 가능한 범위: 비로그인 상태에서는 로컬 스냅샷 기반으로 사용할 수 있고, 로그인 상태에서는 서버 bootstrap 이후 DB 스냅샷을 불러오고 저장할 수 있습니다.
- 현재 진행 중: 사이드바 토글 UX 자연화, 클라이언트 부트스트랩 로딩 체감 개선, 데스크톱 고정 사이드바와 모바일 드로어 상태 분리

## 주요 기능

- 탭 기반 작업 분류
  운동, 공부, 생활, 게임 루틴처럼 성격이 다른 작업을 탭으로 나눠 관리할 수 있습니다.
- 일반 작업 / 반복 작업 분리
  한 번 끝내는 작업과 주기적으로 다시 해야 하는 작업을 다른 모델로 관리합니다.
- 반복 작업 주기 설정
  매일, 매주, 매월, 매년, N일마다 형태로 반복 주기를 지정할 수 있습니다.
- 다중 체크 지원
  반복 작업마다 한 사이클에서 몇 번 체크해야 하는지 목표 횟수를 둘 수 있습니다.
- 시간 기준 초기화
  `plain` 기준과 브라우저 타임존 기준을 나눠 반복 작업 초기화 시점을 설정할 수 있습니다.
- 자동 리셋
  반복 작업은 현재 사이클을 계산해 체크 상태와 완료 시점을 자동으로 초기화합니다.
- 작업 / 탭 순서 변경
  `dnd-kit` 기반 드래그 앤 드롭으로 탭과 작업 순서를 바꿀 수 있습니다.
- 수정 / 삭제 / 복구
  작업과 탭 모두 수정할 수 있고, 삭제 후 짧은 시간 안에 복구할 수 있습니다.
- 인증 후 서버 저장
  Google OAuth 로그인 이후 Supabase 세션과 Prisma를 통해 사용자 프로필과 트래커 스냅샷을 관리합니다.
- 반응형 웹
  모바일에서는 사이드바를 접고 열 수 있고, 데스크톱에서는 고정형 레이아웃으로 사용할 수 있게 구성했습니다.

## 기술 스택

- 프레임워크: Next.js 16, React 19
- 언어: TypeScript
- 스타일링: Tailwind CSS 4
- UI 프리미티브: Radix UI, shadcn/ui
- 상태 관리: Zustand
- 드래그 앤 드롭: dnd-kit
- 날짜 / 시간 처리: `@js-temporal/polyfill`
- 인증: Supabase Auth
- DB / ORM: Prisma, PostgreSQL
- 알림 UI: Sonner
- 테스트: Vitest

## 기술 선택 이유

- Next.js 16
  App Router를 기준으로 인증 흐름, 라우트 그룹, 메타데이터, 서버 액션을 함께 관리하기 좋았습니다.
- Zustand
  탭, 작업, 사이드바처럼 도메인이 다른 상태를 가볍게 분리할 수 있고, hydrate/save 흐름과 붙이기에도 부담이 적었습니다.
- Temporal polyfill
  월말, 윤년, 시간대 같은 경계 조건을 직접 다뤄야 해서 `Date`보다 안전한 계산 모델이 필요했습니다.
- dnd-kit
  단순 정렬 기능에 필요한 접근성과 확장성을 함께 가져가기 좋았습니다.
- Supabase Auth + Prisma
  인증과 세션은 Supabase에 맡기고, 앱 데이터는 Prisma로 명시적으로 다루는 구조가 요구사항에 잘 맞았습니다.
- Radix UI / shadcn/ui / Tailwind CSS
  기본 접근성을 갖춘 프리미티브 위에서 빠르게 UI를 조정하고, 필요한 디자인 디테일을 직접 제어하기 좋았습니다.
- Vitest
  날짜 계산과 반복 작업 동기화처럼 순수 함수 중심 로직을 빠르게 검증하기 좋았습니다.

## 아키텍처 / 폴더 구조

```text
src
├─ actions
│  ├─ auth.actions.ts
│  ├─ profile.actions.ts
│  └─ tracker.actions.ts
├─ app
│  ├─ (auth)
│  ├─ (tracker)
│  ├─ globals.css
│  └─ layout.tsx
├─ features
│  └─ tracker
│     ├─ components
│     │  ├─ sidebar
│     │  └─ task
│     ├─ hooks
│     └─ model
└─ lib
```

- `app/(tracker)`
  실제 트래커 화면 진입점입니다. `(tracker)` 레이아웃에서 `TrackerPersistenceProvider`를 감싸 부트스트랩과 저장 흐름을 붙입니다.
- `app/(auth)`
  로그인 화면과 OAuth 콜백 경로를 담당합니다.
- `actions`
  `bootstrapTracker`, `saveTrackerSnapshot`, `getProfile`처럼 인증/DB와 직접 맞닿는 서버 액션을 모아둔 레이어입니다.
- `features/tracker/components`
  `Header`, `Sidebar`, `TaskList`, `TaskItem`, `TaskDialog`처럼 화면을 구성하는 UI를 모아두었습니다.
- `features/tracker/model`
  타입, Zustand 스토어, 로컬 스토리지 입출력, 반복 작업 계산, persistence provider를 모아둔 도메인 레이어입니다.
- `TrackerPersistenceProvider`
  인증 상태 변경 감지, 로컬 스냅샷 읽기, 서버 bootstrap, 저장 예약, 페이지 이탈 전 flush, 반복 작업 주기 sync를 담당합니다.
- `repeatTask.ts`
  현재 사이클 계산, 다음 초기화 시점 계산, 반복 작업 동기화를 컴포넌트 밖 순수 함수로 분리한 핵심 로직입니다.

## 개발 연혁

1. 초기 구조 세팅과 스토어/저장소 분리
   폴더 구조를 정리하고 `tabStore`, `tabStorage`, hydrate 흐름을 분리했습니다. 화면보다 상태 구조를 먼저 다듬는 방향을 초기에 잡았고, 이후 대부분의 기능 확장은 이 기반 위에서 진행됐습니다.

2. 반복 작업 모델, Temporal 계산, 테스트 강화
   `repeatTask.ts`를 중심으로 반복 작업의 사이클 계산, 다음 리셋 시점 계산, 완료 상태 초기화를 정리했습니다. 월말, 윤년, 시간대 경계 조건 때문에 테스트를 먼저 고도화했고, 현재 저장소의 자동 테스트도 이 축에 집중되어 있습니다.

3. 탭/작업 UI와 삭제 복구, DnD 개선
   `TaskList`, `TaskItem`, `TaskDialog`를 중심으로 CRUD UI를 확장했고, 삭제 후 복구 시 기존 순서를 최대한 보존하도록 position 처리도 보강했습니다. 드래그 중 카드가 찌그러지는 문제, 드래그 핸들 분리, 액션 버튼 배치 같은 디테일도 이 단계에서 많이 다듬었습니다.

4. 반응형 사이드바와 드롭다운, 모바일 대응
   모바일 햄버거 사이드바, `md` 이하 접힘 처리, `TabItem` 드롭다운 메뉴, `MoreHorizontal` 계열 조작 UI를 붙였습니다. 이 과정에서 사이드바는 기능적으로는 완성도가 높아졌지만, 데스크톱/모바일 상태를 함께 관리하는 구조가 후반 UX 이슈의 씨앗이 되기도 했습니다.

5. Supabase / Prisma, Google OAuth, DB 저장
   Supabase Auth와 Prisma를 연결하고, 로그인 후 사용자 프로필과 트래커 스냅샷을 저장하는 흐름을 만들었습니다. `bootstrapTracker`로 초기 스냅샷을 고르고, `saveTrackerSnapshot`으로 변경분을 저장하며, 인증 변화 시 bootstrap을 다시 수행하도록 정리했습니다.

6. 로딩 / skeleton / 사이드바 UX 다듬기와 현재 미완 이슈
   메인 화면 등장 모션을 줄이고 skeleton UI를 도입했으며, 부트스트랩 중 빈 화면을 줄이기 위한 구조 변경을 검토했습니다. 최근 작업의 중심은 기능 추가보다 로딩 체감과 사이드바 토글 UX를 자연스럽게 만드는 데 맞춰져 있고, 이 부분은 아직 진행 중입니다.

## 트러블슈팅 / 설계 판단

- 월말 / 윤년 / 시간대 경계 계산
  반복 작업은 단순히 날짜를 더하는 수준으로 끝나지 않았습니다. `Temporal` 기반으로 현재 사이클과 다음 리셋 시점을 분리 계산해 월말과 윤년, 시간대 차이에서 발생하는 꼬임을 줄였습니다.
- 삭제 후 복구 시 순서 보존
  작업이나 탭을 복구할 때 기존 `position`이 깨지지 않도록, 복구 시점에 다른 아이템의 위치를 다시 밀어 넣는 방식으로 정렬 문제를 해결했습니다.
- stale activeTab 처리
  활성 탭이 삭제된 뒤에도 이전 ID를 참조하는 문제가 생길 수 있어, 탭 상태 normalize와 선택자에서 유효한 활성 탭만 반환하도록 보완했습니다.
- bootstrap 이후 동일 스냅샷 저장 생략
  서버 bootstrap으로 받은 스냅샷과 현재 스냅샷이 같다면 바로 다시 저장하지 않도록 원본 snapshot 비교 로직을 추가했습니다.
- 클라이언트 부트스트랩 로딩 처리
  현재 로딩 이슈는 단순 `loading.tsx`보다 클라이언트 부트스트랩과 hydrate 타이밍에 가깝습니다. 그래서 전체 셸 차단보다 탭 목록, 작업 목록, 헤더 영역 단위의 지연 렌더가 더 적합하다고 판단하고 있습니다.
- 사이드바 UX는 상태 책임 분리 문제
  최근 가장 많이 부딪힌 문제는 사이드바 버튼 그 자체보다도, 데스크톱 고정 사이드바와 모바일 오버레이 드로어를 같은 상태 모델로 다루고 있다는 점이었습니다. 현재 구조는 작동은 하지만, UX 완성도를 높이려면 상태 책임을 다시 나누는 리팩터링이 필요합니다.

## 현재 남은 이슈

- 데스크톱 고정 사이드바와 모바일 오버레이 드로어 상태가 아직 충분히 분리되지 않았습니다.
- 사이드바 닫기 버튼과 헤더 열기 버튼 전환이 완전히 자연스럽지 않습니다.
- 트래커 초기 부트스트랩 로딩은 개선 중이며, 전체 셸 차단보다 탭 목록/작업 목록 단위 지연 렌더가 더 적합하다고 보고 있습니다.
- 최근 작업은 기능 부족을 메우는 단계보다, 이미 구현된 기능의 UX 완성도를 끌어올리는 단계에 가깝습니다.

## 회고

이번 프로젝트에서 가장 잘 맞았던 흐름은 UI를 먼저 찍어내는 방식이 아니라, 탭과 작업의 상태 모델을 먼저 만들고 그 위에 화면을 얹는 방식이었습니다. 덕분에 반복 작업 로직, 삭제 복구, 저장 흐름, 인증 이후 bootstrap 같은 기능을 붙일 때도 핵심 로직을 크게 흔들지 않고 확장할 수 있었습니다.

특히 `Temporal`과 테스트를 통해 날짜 경계 조건을 다룬 경험은 이 프로젝트의 큰 자산이 되었습니다. 반대로 최근 사이드바 UX 작업은 작은 UI 디테일처럼 보여도 상태 책임과 애니메이션 구조가 얽히면 금방 복잡해질 수 있다는 점을 분명하게 보여줬고, 다음 단계에서는 데스크톱과 모바일 사이드바 모델을 더 명확히 분리하는 리팩터링이 필요하다는 교훈도 남겼습니다.
